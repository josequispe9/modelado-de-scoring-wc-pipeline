Sí, es perfectamente posible y es un setup muy común. Te explico cómo funciona la comunicación y dónde está el punto de fricción.

Arquitectura de lo que tenés
Windows Host
├── Docker Desktop (corre contenedor PostgreSQL)
│   └── puerto 5432 expuesto → localhost:5432 (en Windows)
└── WSL2 Ubuntu
    └── script Python → necesita llegar a PostgreSQL
El problema es que WSL2 y Windows no comparten el mismo localhost. WSL2 tiene su propia IP interna.

Cómo conectarse: las opciones
Opción 1 — Usar la IP del host Windows desde WSL2 (la más simple)
Desde dentro de WSL2, Windows es accesible via una IP especial. En versiones recientes de WSL2:
bash# Dentro de Ubuntu/WSL2
cat /etc/resolv.conf  # la IP del "nameserver" es tu Windows host
O directamente usar el hostname mágico:
python# En tu script Python
conn = psycopg2.connect(
    host="host.docker.internal",  # ← esto resuelve al host Windows
    port=5432,
    database="tu_db",
    user="postgres",
    password="tu_password"
)

host.docker.internal funciona desde WSL2 en la mayoría de configuraciones modernas de Docker Desktop.

Opción 2 — Exponer el puerto y conectar vía IP dinámica de WSL2
bash# Desde WSL2, obtener la IP del host
ip route show | grep default | awk '{print $3}'
Usar esa IP en el script Python.
Opción 3 — Poner el script Python también en Docker (la más limpia)
Si metés tu script en un contenedor dentro de la misma red de Docker:
pythonhost="postgres"  # nombre del servicio en docker-compose

Configuración de PostgreSQL (el contenedor)
Por defecto PostgreSQL solo escucha conexiones locales. Tenés que asegurarte de que el contenedor:
1. Expone el puerto:
yaml# docker-compose.yml
services:
  postgres:
    image: postgres:15
    ports:
      - "5432:5432"  # ← esto es clave
    environment:
      POSTGRES_PASSWORD: password
      POSTGRES_USER: postgres
      POSTGRES_DB: midb
2. Acepta conexiones externas — con la imagen oficial de Docker esto ya viene configurado por defecto (listen_addresses = '*'). No necesitás tocar pg_hba.conf si usás la imagen oficial.
Podés verificarlo:
bashdocker exec -it <container_name> psql -U postgres -c "SHOW listen_addresses;"
Debería devolver *.

Este error específico significa que el TCP handshake falla antes de que PostgreSQL siquiera vea la conexión. No es un problema de credenciales ni de pg_hba.conf. Es de red o de puerto.

Diagnóstico paso a paso
Paso 1 — ¿El contenedor está corriendo y el puerto expuesto?
bash# En Windows PowerShell o CMD
docker ps
Buscá que diga algo como 0.0.0.0:5432->5432/tcp. Si dice solo 5432/tcp sin el 0.0.0.0, el puerto no está expuesto al host.

Paso 2 — ¿Windows escucha en ese puerto?
powershell# En PowerShell (Windows)
netstat -an | findstr 5432
Tenés que ver algo así:
TCP    0.0.0.0:5432    0.0.0.0:0    LISTENING
Si no aparece nada → el contenedor no está exponiendo el puerto correctamente.

Paso 3 — ¿WSL2 puede llegar al host?
bash# En Ubuntu/WSL2
ping host.docker.internal
Si no resuelve ese hostname, probá con la IP directa:
baship route show | grep default | awk '{print $3}'
# Anotá esa IP (algo como 172.x.x.x) y usala en lugar de host.docker.internal

Paso 4 — Probar la conexión TCP pura
bash# En WSL2
nc -zv host.docker.internal 5432

# Si nc no está instalado:
python3 -c "
import socket
try:
    s = socket.create_connection(('host.docker.internal', 5432), timeout=3)
    print('CONEXIÓN OK')
    s.close()
except Exception as e:
    print(f'FALLO: {e}')
"

1. El problema central que hay que identificar primero
Antes de proponer soluciones, un Senior debería nombrar explícitamente el problema:

"El issue raíz es que WSL2, Docker Desktop y el host Windows operan en namespaces de red distintos. Cualquier solución que dependa de IPs dinámicas o de localhost asumiendo un entorno único va a ser frágil."

Eso solo ya diferencia a alguien que entiende el problema de alguien que memorizó el fix.

2. La estrategia — tres pilares
Pilar 1: Todo dentro de Docker (eliminar la fricción)
La solución más robusta es no depender de WSL2 como entorno de ejecución de servicios. Meter todo en contenedores dentro de la misma red Docker:
yamlservices:
  postgres:
    image: postgres:15
    networks:
      - backend

  mi_script:
    build: .
    environment:
      DB_HOST: postgres  # resolución por nombre de servicio
    networks:
      - backend

networks:
  backend:
    driver: bridge
Dentro de esta red, postgres resuelve por DNS interno de Docker. No hay IPs dinámicas, funciona igual en local y en CI/CD.

Pilar 2: Variables de entorno para la configuración de conexión
Nunca hardcodear el host. El script Python recibe la config del entorno:
pythonimport os
import psycopg2

conn = psycopg2.connect(
    host=os.getenv("DB_HOST", "localhost"),
    port=os.getenv("DB_PORT", 5432),
    database=os.getenv("DB_NAME"),
    user=os.getenv("DB_USER"),
    password=os.getenv("DB_PASSWORD")
)
Localmente usás un .env. En CI/CD usás secrets del pipeline. El código no cambia.

Pilar 3: Docker network explícita y nombrada
No depender de la red default que Docker crea automáticamente. Definir la red explícitamente en docker-compose.yml como arriba. Esto hace que el comportamiento sea predecible y declarativo.

3. Para el caso donde WSL2 es inevitable
Si por alguna razón el script tiene que correr en WSL2 (entorno virtual existente, dependencias nativas, etc.):
bash# En WSL2 esto siempre resuelve al host Windows
host.docker.internal
Y configurarlo como variable de entorno en el .env de WSL2:
bashexport DB_HOST=host.docker.internal
Pero aclarar que esto es un workaround, no una arquitectura.

4. CI/CD — cómo se traslada
En GitHub Actions / GitLab CI, el equivalente es un service container:
yaml# GitHub Actions
services:
  postgres:
    image: postgres:15
    env:
      POSTGRES_PASSWORD: password
    ports:
      - 5432:5432
Y el script se conecta a localhost:5432 porque en CI todo corre en la misma red del runner. Las variables de entorno hacen que el mismo código funcione en ambos contextos sin modificaciones.
